What Is This Tool?

This tool helps you figure out where to place your infrastructure ("cells") around the world so that your endpoints get fast, reliable connections.

You tell the tool:

  • Where your existing cells are today
  • Where your endpoints are located
  • What latency (in milliseconds) you promised them in your SLA
  • Optionally, how much your current cells cost per month

The tool then uses realistic distance parameters and a 32x32 AWS latency matrix to recommend exactly where to deploy new cells — optimizing to strictly meet your SLA for the lowest possible cost.

Key Concepts

Cell

A server or piece of infrastructure you deploy in an AWS data center.

Endpoint

An integration partner, service consumer, or client application that needs to connect to your cell.

SLA (Service Level Agreement)

The maximum latency (in milliseconds) you've promised your endpoints.

Latency

The round-trip time (in milliseconds) for data to travel from your cell to your endpoint and back.

1 Current Footprint

Tell the tool where you already have cells deployed. This is optional.

Use List View to browse regions by continent, or Map View to visualize your selections.

Below the cell selection, you'll find the Infrastructure Cost Profile accordion — see the next section for details.

Cost Calculator

The cost calculator is optional but powerful. It lets you estimate the monthly OPEX impact of deploying new cells.

The "Hidden Global Index"

The tool has a hardcoded AWS Regional Price Index anchored to US East (N. Virginia) = 1.00x. Every other region is priced relative to it:

US East (Virginia)1.00x
Europe (Paris)1.10x
Asia Pacific (Tokyo)1.25x
South America (São Paulo)1.50x
Africa (Cape Town)1.30x
Middle East (Dubai)1.25x

Reverse-Engineering the Global Base

When you input a cost for Paris ($2,674), the tool knows Paris is 1.10x. It reverse-engineers the normalized base:

$2,674 ÷ 1.10 = $2,430 (True Global Base)

Blended vs. Specific Baseline

  • Blended: Averages reverse-engineered base costs from all inputted cells.
  • Specific: Anchors to one specific cell of your choice.

2 Endpoint Locations

Add endpoints via the AWS Regions tab or World Cities tab. You can also click anywhere on the map to snap to the nearest major city.

3 SLA Configuration

Latency Estimation

Choose how the tool calculates latency between your infrastructure and end users:

  • Naive Mode: Uses a simplified "proxy" approach. If an endpoint is in a non-AWS city, the tool finds the geographically closest AWS region to them and uses the known AWS backbone latency from your cell to that proxy region. It ignores exact fiber distances and local internet infrastructure quality.
  • Realistic Mode (recommended) introduces an intelligent hybrid distance model:
    • Direct Distance Model: For non-AWS locations (like world cities), the matrix is bypassed in favor of a direct Haversine formula distance calculation combined with a 0.012 ms/km fiber speed penalty.
    • Infrastructure tax: Adds penalty latency based on the tier of the city (Tier 1: +5ms · Tier 2: +20ms · Tier 3: +40ms · Tier 4: +60ms).

4 Coverage Analysis

The dashboard shows your total coverage, pending coverage, number of new cells needed, and estimated monthly OPEX.

You can also use the Latency Explorer to debug exactly why a cell passed or failed for a given endpoint.

How It Works Under the Hood

Cost-Effective Coverage Algorithm

The tool strictly treats the SLA as a pass/fail gate. For each candidate AWS region, the tool calculates a Cost Efficiency Score:

Score = Region_Cost ÷ Endpoints_Covered

Lower scores win. The algorithm iteratively picks the cheapest combination of regions to get 100% of your un-covered endpoints into the "Pass" state.

Direct Distance vs AWS Backbone

When computing latency, the tool checks the entity type of both endpoints:

  • AWS to AWS: Uses a hardcoded 32x32 ping matrix representing the true latency of the AWS global backbone.
  • AWS to City (Naive Mode): Finds the nearest AWS region to the target city and uses the 32x32 backbone matrix latency directly to that proxy region.
  • AWS to City (Realistic Mode): Defaults to the Direct Distance Model. It calculates the exact Haversine geographic distance between the data center and the city coordinates, multiplies it by 0.012 ms per kilometer, and adds a last-mile infrastructure tier tax.

Why 0.012 ms/km? (The Physics)

This propagation delay factor is mathematically grounded in the physics of fiber optic networking:

  • Speed of light in a vacuum: ~300,000 km/s.
  • Speed of light in glass (fiber): ~200,000 km/s (due to a refractive index of ~1.5).
  • Time per kilometer: 1 km / 200,000 km/s = 0.000005 seconds, or 0.005 ms per km.
  • Round-Trip Time (RTT): Networking latency measures the time to go and come back. Doubling the one-way time: 0.005 ms × 2 = 0.01 ms per km.

What Is This Tool?

The Onboarding Cost Calculator scopes out the engineering hours and timeline required to bring a new customer online.

It acts as an "MS Project" style interactive grid, generating a Financial Quote (Internal Cost + Customer Price) and an Interactive Project Timeline all in one unified view.

Deployment Scopes (Tiers)

The onboarding effort scales based on the infrastructure and integration requirements for the specific customer. Select the appropriate tier from the top ribbon to load the default task template.

  • Tier 1: Full Deployment (New Cell + New Provider)
    The heaviest lift. Requires standing up new AWS infrastructure, building initial API handshakes with a new service provider, and running full security validation.
  • Tier 2: Integration Only (Existing Cell + New Provider)
    Uses existing infrastructure in the region, but requires establishing API handshakes and security posture with a new service provider.
  • Tier 3: Standard Onboarding (Existing Cell & Provider)
    The lightest workload. The cell and provider are already established. Focuses purely on QA, training, and cutover for the new end customer.
Warning: Resetting Timeline
Changing the Deployment Scope tier will ask for confirmation before resetting your entire timeline to the default template for that tier. Save your work (Export JSON) before switching if you want to keep custom tasks!

How the Math Works

The 4-Hour Ceiling Rule

Project estimation must be realistic. If you type "17" hours into the grid, the calculator automatically snaps it to 20 hours. We strictly enforce a 4-hour (half-day) minimum increment for all scoping to avoid messy fractions.

Financial Formulas

Internal Cost = Total Hours × Blended Hourly Rate
Customer Price = Internal Cost ÷ (1 - Margin %)

Interactive Gantt Features

The unified grid on the left and the chart on the right are perfectly synced.

  • Drawing Dependencies: Hover over a blue task bar. Grab the small circle at the end of the bar and drag it to the start of another task to draw a dependency arrow.
  • Deleting Dependencies: Double-click on any green dependency arrow to delete it.
  • Generic Calendar: The top scale abstracts real-world dates (ignoring weekends) into clean "Day 1, Day 2" progression blocks for pristine scoping.

Dragging to Resize Tasks

You can grab the edge of any blue task bar and stretch it. What happens next depends on the Link Duration to Cost checkbox in the top ribbon:

Linked (Checked)
Stretching a task from 2 days to 3 days adds billable hours (e.g., jumps from 16h to 24h) and automatically updates the input grid. Your total project cost and invoice price will immediately increase.
Unlinked (Unchecked)
Stretching a task from 2 days to 3 days simply means the resource is working part-time. The billed hours and financial cost remain perfectly locked.
